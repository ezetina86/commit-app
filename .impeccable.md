## Design Context

### Users
A developer (the creator) using Commit as a personal daily habit tracking tool. The context is a terminal/code environment — used during or after a coding session, likely on a laptop at a desk. The user is technical, disciplined, and data-oriented. No onboarding needed; the interface is built for one person who understands it intimately.

### Brand Personality
**Obsessive, technical, satisfying.** Using Commit should feel like running a well-crafted CLI script — precise outputs, no fluff, immediate feedback. The interface rewards consistency and makes the user feel like a high-performer tracking their own systems. The satisfaction is in the data, not the decoration.

### Aesthetic Direction
Terminal-native dark UI. GitHub contribution graph greens (`#0E4429` → `#39D353`) layered on near-black backgrounds (`#0B0E14`). JetBrains Mono throughout — no exceptions. Bracket decorators, CLI prompts (`root@commit-sys:~#`), typewriter animations, CRT scanline effects. No decorative elements that don't serve the terminal metaphor.

**Anti-references**: glassmorphism, gradient text, soft rounded cards, pastel palettes, "product" design patterns.
**Nearest references**: GitHub contribution graph, terminal dotfiles configs, htop, a well-tuned .vimrc.

### Design Principles
1. **Terminal-first**: Every element should feel at home in a CLI context. Use `>`, `$`, `[brackets]`, and code-style labels as visual language.
2. **Signal over noise**: Data density is a feature. The grid is the hero. Never add decoration that competes with the contribution data.
3. **Monochrome discipline**: One accent family (the green spectrum). No additional hue families except destructive actions (red). Color earns its place.
4. **Earned feedback**: Interactions feel like commands being processed. Transitions are state changes, not celebrations. 150–200ms, ease-out.
5. **Obsessive precision**: Every spacing value, color, and alignment should map to the token system. Hardcoded values are a sign of drift.
